# Staging OpenAI Mentor Runtime

This runbook defines the staging contract for enabling the LionsForge AI mentor through the OpenAI Responses API without exposing credentials.

## Existing runtime

The backend mentor provider already supports:

- the OpenAI Responses API
- strict JSON-schema output validation
- bounded retries and request timeouts
- local provider health state
- deterministic fallback when the provider is disabled or unavailable

## Required staging configuration

Create a GitHub environment secret in the `staging` environment named `STAGING_OPENAI_API_KEY`.

The deployment workflow must map that environment secret into the Kubernetes backend secret as `OPENAI_API_KEY`. The value must never be printed, echoed, written to artifacts, committed, or included in logs.

The backend deployment must continue to receive the existing database and JWT settings. OpenAI configuration is additive and must not replace those values.

## Deployment acceptance

After deployment, verify all of the following:

1. Provider health reports OpenAI enabled and configured.
2. An authenticated Mentor request returns a schema-valid generated response.
3. The response contains answer, reasoning, assumptions, confidence, alternative viewpoints, and recommendations.
4. No credential material appears in application logs, workflow logs, Kubernetes events, or artifacts.
5. A simulated provider timeout or connection failure returns the deterministic mentor fallback instead of a 5xx response.
6. Removing or invalidating the staging credential changes provider health to disabled or misconfigured without taking the API offline.

## Security rules

- Use a project-scoped runtime key rather than a personal general-purpose key.
- Store the key only in the GitHub `staging` environment and the Kubernetes Secret generated by the deploy workflow.
- Do not place the key in repository variables, plaintext manifests, `.env` files committed to source control, issue comments, pull-request descriptions, or chat messages.
- Rotate the key after any suspected exposure.
- Keep staging and production credentials separate.

## Exit criteria

Issue #81 can be closed only when the deployment workflow safely references the staging secret, staging provider health reports enabled, generated Mentor responses pass schema validation, and fallback behavior remains operational during a simulated provider failure.
